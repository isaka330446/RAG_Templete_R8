仕様書 00: 全体サマリ

国税庁TaxAnswer、通達、質疑応答事例、関連PDFを対象にした相続税・贈与税向けRAGアプリの最新版仕様です。

目的

  • 利用者が税務文書を根拠付きで検索し、RAG回答と根拠本文を確認できるようにする。
  • 管理者が回答品質、根拠なし、低信頼、報告、承認済みQA、aliasを継続改善できるようにする。
  • 開発者がJSONL/CSV評価セットでバッチ推論、自動評価、手動採点を実行できるようにする。
  • 会社Windowsオフライン環境を前提に、URLとSQLite設定を一元管理する。

現行アプリ構成

区分実装起動
RAG APIFastAPI。検索、生成、QAキャッシュ、ログ、管理API。python scripts/run_api.py
ユーザー向けチャットFastHTMLモダンUIのみ。根拠は大型ポップアップ。python -m app_fasthtml_modern.main
管理者アプリFastHTML RAG改善コンソール。6画面構成。python -m app_fasthtml_admin_common.main
開発者評価UIFastHTML。バッチ推論、自動評価、手動採点。python -m app_dev_eval.main

重要な設計方針

  • URL、host、port、LLM/Embedding/Reranker接続先、CORS許可URLは config/settings.json の urls に集約する。
  • .env は原則 SETTINGS_PATH など最小限に限定する。
  • SQLiteは本番必須。logging.required=true と answer_cache.required=true が標準。
  • SQLite標準の journal_mode は DELETE。これはDB削除ではなくrollback journal方式。
  • 一般ユーザー向けフロントは app_fasthtml_modern/main.py の1系統のみ。
  • 承認済みQA本体は approved_qa、検索入口は approved_qa_aliases。
  • ユーザー質問を毎回LLMで拡張しない。実行時はEmbedding 1回 + alias検索を基本にする。
  • SearchTagは削除せず補助機能として扱い、主改善導線はQA alias、承認QA化、文書不足確認を優先する。

廃止済み

  • 旧ユーザー画面、旧管理画面、旧評価画面はFastHTML構成へ統合済み。
  • ポップアップ用ラッパーアプリは廃止。ユーザー向けFastHTML本体がポップアップ方式を標準採用。
  • 根拠表示のside drawer切替、UIモード切替環境変数は廃止。
